# Aetna v5.3 — Plan

## Where v5.3 fits

v5.0 split the codebase into `aetna-{core,wgpu,demo}`. v5.1 decoupled text from glyphon (cosmic-text + swash + core-owned `GlyphAtlas`). v5.2 carried the wgpu backend onto wasm, and along the way validated that the core/backend boundary survives one platform port.

v5.3 is the second backend. The whole point of the v5.0 split was that "core knows about a tree of `DrawOp`s, the backend knows about pipelines and pixels" — and that contract has been load-bearing since v5.0 but has only ever been tested by *one* backend (wgpu). v5.3 puts that contract under load by reproducing the same draw stream through a wholly separate GPU API: Vulkan, via [`vulkano`](https://github.com/vulkano-rs/vulkano), with WGSL transpiled to SPIR-V via [`naga`](https://github.com/gfx-rs/wgpu/tree/trunk/naga) (the same crate `wgpu` ships internally).

This document covers **v5.3 only**. v5.4 (Vulkano parity across all v0.4 demos + extraction of any shared runner shape) gets its own plan once v5.3 lands.

## v5.3 — first slice

### Goal

Stand up a parallel `aetna-vulkano` backend that reproduces **one** v0.4 fixture end-to-end through Vulkan. Specifically: `cargo run -p aetna-vulkano-demo --bin counter` produces the same interactive counter we get from `cargo run -p aetna-demo --bin counter`, with the same hover/press/focus visuals, same fonts, same hit-test behaviour. The wgpu backend keeps working unchanged.

The narrow target — **one fixture, native only** — is deliberate. v5.3 is asking *does the architecture hold across two GPU APIs?* It is not asking *is the second backend feature-complete?* (that's v5.4).

### Acceptance criteria

- `cargo build --workspace` succeeds with the new `aetna-vulkano` and `aetna-vulkano-demo` crates added.
- `cargo run -p aetna-vulkano-demo --bin counter` opens a window rendering the counter App with: rounded buttons, hover/press envelopes, click-to-increment, focus ring on Tab.
- A side-by-side visual A/B against `cargo run -p aetna-demo --bin counter` looks indistinguishable to the eye (same colors, same metrics, same fonts).
- `cargo test --workspace --lib` continues to pass (no regression in `aetna-core` or `aetna-wgpu`).
- The `wasm-pack build --target web` path for `aetna-web` still succeeds.
- `aetna-core` is unmodified except for additive changes (e.g., a small WGSL-validation helper if needed). The point of v5.3 is to land a backend, not to rework core.

### What changes, what doesn't

| Crate | v5.3 change |
|---|---|
| `aetna-core` | unchanged (or additive only — see "Open questions" below) |
| `aetna-wgpu` | unchanged |
| `aetna-demo` | unchanged |
| `aetna-web` | unchanged |
| `aetna-vulkano` | **new** — Vulkan backend, peer to `aetna-wgpu` |
| `aetna-vulkano-demo` | **new** — winit + vulkano harness with the counter bin |

The deliberate non-decision: v5.3 does **not** introduce a `Painter` trait, does **not** move `Runner` into `aetna-core`, does **not** rework `aetna-demo` to be backend-agnostic. The wgpu-side `Runner` and the new vulkano-side `Runner` are siblings. Once both exist and have been used, v5.4 evaluates whether a shared trait is worth extracting; designing it before seeing the second backend's actual shape is premature.

## The boundary aetna-vulkano has to honour

aetna-wgpu's external surface (per the v5.2 codebase survey) is the contract the vulkano peer must reproduce. The Vulkan-flavoured method signatures will differ in their GPU-typed parameters (`vulkano::device::Device` instead of `wgpu::Device`, etc.) but the *shape* — what the host calls, what gets returned, when — is the same.

| Method | Purpose | GPU-typed args become |
|---|---|---|
| `Runner::new(device, queue, target_format)` | bring up pipelines + atlas mirror | `Arc<vulkano::device::Device>`, `Arc<vulkano::device::Queue>`, `vulkano::format::Format` |
| `set_surface_size(w, h)` | propagate framebuffer extent | unchanged |
| `register_shader(device, name, wgsl)` | install a custom shader pipeline | naga compiles WGSL → SPIR-V; vulkano builds a graphics pipeline against the standard `QuadInstance` layout |
| `prepare(device, queue, &mut tree, viewport, scale)` | layout + animation tick + draw-stream build + GPU upload | uploads write into vulkano `Buffer`s and `Image`s |
| `draw(pass)` | record draw calls | takes a vulkano command-buffer builder instead of a `wgpu::RenderPass` |
| `pointer_*` / `key_down` / `set_hotkeys` / `ui_state` / `rect_of_key` / `debug_summary` | input plumbing + introspection | GPU-agnostic; identical signatures |

The paint-stream loop is identical: walk `DrawOp[]`, dispatch `Quad`/`GlyphRun`/`BackdropSnapshot`. Scissor-run batching by `(scissor, pipeline, page)` is identical. The text path is structurally identical (mirror `GlyphAtlas` into per-page GPU textures, sync dirty regions during `prepare`, bind per-page during `draw`). Only the GPU-API calls change.

## Migration order (sequence of commits)

Each step compiles cleanly and (where applicable) renders something. Don't accumulate broken intermediate state.

1. **Workspace scaffold.** Add `crates/aetna-vulkano/` and `crates/aetna-vulkano-demo/` with empty libs and `Cargo.toml`s wired into the workspace. `cargo build --workspace` succeeds.
2. **WGSL → SPIR-V helper.** Small module in `aetna-vulkano`: `wgsl_to_spirv(name: &str, source: &str) -> Vec<u32>`. Use `naga::front::wgsl::parse_str` → `naga::back::spv::write_vec`. Unit-test against `stock_wgsl::ROUNDED_RECT` and `stock_wgsl::TEXT`.
3. **vulkano + winit hello-triangle bring-up.** In `aetna-vulkano-demo`, open a window, create a Vulkan instance + surface + swapchain + render-pass + clear-color frame loop. No Runner yet. This is the "is our vulkano + winit 0.30 stack alive?" check.
4. **Runner skeleton — input + state.** Implement `Runner::new`, the input methods (`pointer_*`, `key_down`, `set_hotkeys`), and `ui_state()` / `rect_of_key()`. These are GPU-agnostic; copy from `aetna-wgpu` verbatim. No GPU resources yet. `prepare()` returns an empty `PrepareResult`.
5. **Rounded-rect pipeline.** Build a vulkano `GraphicsPipeline` from the SPIR-V output of step 2. Match the `QuadInstance` vertex layout exactly (4 × `vec4` slots per the survey). Implement enough of `prepare()` to walk `DrawOp::Quad` runs and pack instances; implement `draw()` to record the resulting draw calls into the host's command buffer. At this point the counter renders *without text* — solid rounded buttons.
6. **Text pipeline.** Mirror `text_atlas::GlyphAtlas` into per-page vulkano `StorageImage`s + descriptor sets. Compile `text.wgsl` to SPIR-V; build the text pipeline with premultiplied-alpha blend. Implement the dirty-region upload path (matching `aetna-wgpu::text::TextPaint::flush`). Now the counter renders fully — text included.
7. **Custom shader path.** Make `register_shader` compile WGSL → SPIR-V → graphics pipeline at register time, so the same custom-shader contract that `aetna-wgpu` exposes is available. (The counter doesn't exercise this; we still implement it now to keep the Runner surface complete and to exercise the path with a unit test or smoke fixture.)
8. **Acceptance pass.** Side-by-side visual A/B between wgpu and vulkano counter bins. Capture both. If they diverge meaningfully (color, metric, focus-ring shape, glyph anti-aliasing), debug. Document any unavoidable tiny differences (e.g., MSAA defaults).

Steps 1–4 are mechanical scaffolding. Step 5 is the architectural moment — **if `DrawOp` survives a foreign GPU API, the v5.0 boundary is real**. Steps 6–8 are completion.

## Naming + crate layout

```
crates/
├── aetna-core/                     unchanged
├── aetna-wgpu/                     unchanged
├── aetna-demo/                     unchanged
├── aetna-web/                      unchanged
├── aetna-vulkano/                  new — Vulkan peer to aetna-wgpu
│   └── src/
│       ├── lib.rs                    Runner + paint-stream loop (mirrors aetna-wgpu/lib.rs)
│       ├── pipeline.rs               build_quad_pipeline factory; FrameUniforms layout
│       ├── instance.rs               QuadInstance + pack_instance + scissor math (copy from aetna-wgpu)
│       ├── text.rs                   per-page Vulkan image mirror of GlyphAtlas
│       └── naga_compile.rs           wgsl_to_spirv helper
└── aetna-vulkano-demo/             new — winit + vulkano harness
    └── src/
        ├── lib.rs                    run<A: App>(...) — sibling of aetna-demo::run
        └── bin/
            └── counter.rs            re-uses aetna_demo's Counter App, drives via aetna-vulkano
```

`instance.rs` is duplicated from `aetna-wgpu` rather than shared. The two backends agree on the `QuadInstance` *layout* (same 4 × `vec4` ABI, same `pack_instance`); whether to move that file into `aetna-core` is a v5.4 decision once we've seen what diverges in practice.

## Risks to flag

- **vulkano + winit 0.30 compatibility.** Vulkano historically tracks winit closely but versions sometimes lag. If the latest vulkano release doesn't speak winit 0.30, we either pin an older winit *for the vulkano demo only* (ugly but workable since aetna-core is winit-agnostic), or use vulkano's `Surface::from_window` raw-handle path through `raw-window-handle` (cleaner). Investigate at step 3.
- **naga version drift vs `wgpu`.** wgpu 23 ships with a particular naga version; if we depend on a different one we may see WGSL features parse in one and not the other. Mitigate by pinning naga to the same minor that `wgpu = "23"` uses (check `cargo tree`).
- **SPIR-V validation strictness.** Vulkan validates SPIR-V harder than wgpu's WebGL2 backend validates GLSL. If naga emits SPIR-V that vulkano's validator rejects, we may need to enable `naga::valid::Validator` before emit and surface its errors. Add this check inside `wgsl_to_spirv` from day one.
- **Premultiplied alpha conventions.** The text pipeline blends premultiplied. Vulkano's `BlendState` defaults differ from wgpu's; double-check the blend equation matches `aetna-wgpu::text` byte-for-byte.
- **sRGB framebuffer format.** `aetna-demo` picks an sRGB swapchain format and converts the clear color from sRGB to linear (`srgb_to_linear` in `aetna-demo/src/lib.rs`). The vulkano harness must match.

## Out of scope for v5.3

- Full v0.4 demo coverage (scroll_list, hotkey_picker, animated_palette, settings) — v5.4.
- A `Painter` trait or any shared `Runner` abstraction. Two parallel Runners is the v5.3 shape; trait extraction (if it happens) is a v5.4 decision once both Runners are mature.
- Wasm vulkano. Vulkan-on-the-web is not a thing today; this stays native-only.
- Headless render fixtures through vulkano. The wgpu fixtures already serve as the visual reference; replicating them under vulkano is v5.4.
- Performance work. v5.2 settled wasm performance; vulkano's perf profile is whatever it is at v5.3 baseline.
- Touching `aetna-web`. Wasm continues through wgpu; vulkano is an additional native backend, not a replacement.

## Open questions for the implementing instance

1. **Should the demo harness live in `aetna-vulkano-demo` (new crate) or feature-gated inside `aetna-demo`?** Recommend: new crate. Keeps `aetna-demo`'s wgpu surface clean and avoids a workspace-wide `cfg`-juggling exercise. v5.4 can revisit if the duplication smells bad.
2. **Should `instance.rs` (`QuadInstance` + `pack_instance` + scissor math) move into `aetna-core` to be shared by both backends?** The layout is genuinely backend-agnostic. Recommend: hold off in v5.3. Decide in v5.4 with both backends in front of us, alongside the broader Painter-trait question. Premature sharing is the same trap as a premature trait.
3. **vulkano version.** Pick the most recent stable that supports our winit and our naga. Ballpark current is `0.34.x`, but verify at step 1.
4. **Should the WGSL-validation pass run at `register_shader` time or be deferred to pipeline build?** Recommend: at `register_shader`, so the failure surfaces close to the user's `register_shader` call rather than during the next draw. Mirrors how `aetna-wgpu` panics on bad WGSL today.
5. **Do we need a `naga`-side feature gate to avoid pulling SPIR-V tooling into `aetna-wgpu`?** No — `aetna-wgpu` doesn't depend on `aetna-vulkano`, and naga is only a `aetna-vulkano` dep. wgpu pulls in its own internal copy of naga regardless.

## What v5.3 does *not* commit us to

- The Vulkano backend is one peer. If, while building it, we discover the contract `aetna-core` exposes is wrong-shaped for Vulkan in some load-bearing way (e.g., the `draw(pass)` sequencing assumes a wgpu-shaped lifetime that vulkano can't honour), **the right response is to surface that finding and adjust the v5.3 plan**, not to bend the implementation around it. The whole reason we're building a second backend is to learn this.
- The `Painter` trait, if it ever exists, is a v5.4 question. v5.3 explicitly ships two concrete Runners.
- v5.4's scope is still negotiable. If v5.3 reveals the right shape is "shared Runner + thin Painter trait" we'll commit to that in v5.4; if it reveals "two parallel Runners is correct, no trait needed" we'll commit to that instead.
