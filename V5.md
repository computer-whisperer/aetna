# Aetna v5 вЂ” Plan

## Where v5 fits

`attempts/attempt_1` through `attempt_4` each settled a load-bearing premise. v0.4 closed that arc вЂ” rendering substrate, interactive contract, animation system, library/host split are all proven through working demos and 50+ tests. v5 is the architectural cleanup pass. It does not add capabilities. It lands the same capabilities through a structure that holds across two backends (wgpu, vulkano) and two targets (native, wasm).

Once v5 lands, `attempts/` becomes archival. The canonical Aetna codebase lives at the repo root under `crates/`.

## v5 milestones

| Slice | Scope | Premise it settles |
|---|---|---|
| **v5.0** | Crate split, `El` side-map refactor, module split inside core. Glyphon retained. | Does the core/backend boundary survive a clean separation? |
| **v5.1** | Text rendering decoupled. cosmic-text + swash + our atlas in core; backend uploads to its own texture. Glyphon dies. | Can text live above the backend? |
| **v5.2** | wasm target through wgpu (WebGPU + WebGL2). `web-time` for `Instant`. Async font loading. | Does the library hold under no-threads, no-filesystem? |
| **v5.3** | Vulkano backend. Reproduce one fixture native through `aetna-vulkano`. WGSLв†’SPIRV via naga. | Does the core/backend boundary actually hold across two GPU APIs? |
| **v5.4** | Vulkano parity. All v0.4 demos through vulkano native, including text + animation. | Completion. |

This document covers **v5.0 only**. v5.1вЂ“v5.4 each get their own plan as we reach them.

## v5.0 вЂ” first slice

### Goal

Move the v0.4 codebase into a proper crate structure; move library bookkeeping off `El` into `UiState` side maps; split `event.rs` and `tree.rs` into focused modules. **No capability changes.** Every v0.4 demo runs identically.

### Acceptance criteria

- `cargo test --workspace` passes all v0.4 tests (в‰Ґ51).
- All four interactive demos behave identically: `counter`, `scroll_list`, `hotkey_picker`, `animated_palette`.
- All headless renders produce visually identical SVG/PNG artifacts (`out/*.svg` is the reference).
- `attempts/attempt_4/` is **unmodified throughout** вЂ” kept as the baseline until v5 lands.

### The `El` side-map refactor

`El` has 33 fields. Six are *library bookkeeping* the build closure should never set:

| Field | New home |
|---|---|
| `computed: Rect` | `UiState::computed_rects: HashMap<String, Rect>` |
| `scroll_offset_y: f32` (the El copy) | already lives in `UiState::scroll_offsets`; remove the duplicate |
| `hover_amount: f32` | `UiState::envelopes: HashMap<(String, EnvelopeKind), f32>` |
| `press_amount: f32` | same map |
| `focus_ring_alpha: f32` | same map |
| `state: InteractionState` | `UiState::node_states: HashMap<String, InteractionState>` |

`computed_id: String` stays on `El` вЂ” it's the key for the side maps, and there's no way to look it up otherwise without re-walking the tree.

**Net:** `El` goes from 33 в†’ 27 author-set fields + `computed_id` + `children` = 29 fields. The numerical reduction is modest. The semantic win is the point: a fresh `El` from the build closure carries zero library state. Side maps are written by layout/animation passes, read by `draw_ops` and the painter.

**Pass changes:**
- **Layout** writes `computed_rects` keyed by `computed_id`. Doesn't mutate `El.computed` (field gone).
- **`apply_state`** reads envelopes from `UiState`, applies via `Color::mix`. Same logic, different read source.
- **`draw_ops`** takes both `&El` and `&UiState` so it can look up rects + envelopes.
- **Animation tick** reads/writes envelopes in `UiState`, not on `El`.

**Risk to flag:** `(String, EnvelopeKind)` map keys allocate per envelope. For deeply nested trees that's measurable. Mitigations (interned ids, integer keys) exist; defer until profiling shows it.

### Naming

- `UiRenderer` в†’ splits into **`WgpuPainter`** (GPU resources + paint) and **`Runner`** (UiState + event routing + tick).
- `Runner` lives in `aetna-wgpu` for v5.0. v5.3 settles whether it can move to `aetna-core` parameterized over a `Painter` trait.
- Crate names: `attempt_4` в†’ `aetna-core`; `attempt_4_demo` в†’ `aetna-demo`; new `aetna-wgpu`.

### Migration order (sequence of commits)

1. **Workspace scaffold.** Root `Cargo.toml` with `[workspace]`. Empty `crates/aetna-{core,wgpu,demo}`. `cargo build` succeeds.
2. **Promote manifestos.** `SHADER_VISION.md` and `LIBRARY_VISION.md` to root.
3. **Port aetna-core verbatim.** Copy v0.4 source, rename crate to `aetna-core`. No refactor. Tests pass.
4. **Port aetna-wgpu.** Extract `wgpu_render.rs` content, rename `UiRenderer`в†’`WgpuPainter` (paint half) and `Runner` (state half). Internal module split.
5. **Port aetna-demo.** Move bins. Interactive demos work.
6. **Module split inside aetna-core.** `event.rs` в†’ 4 files; `tree.rs` в†’ 3 files. Tests pass.
7. **`El` side-map refactor.** Move bookkeeping fields to `UiState`. Updates layout, `draw_ops`, anim tick, state apply. Biggest commit; tests pass.
8. **Final cleanup.** Update root `README.md` to reflect new structure. Verify all artifacts regenerate.

Steps 1вЂ“5 are mechanical (the goal is making the structure *exist*). Step 6 is mechanical-but-detailed. Step 7 is the architectural step. Each commit must keep tests green and demos working.

### Out of scope for v5.0

- Text rendering changes (glyphon stays вЂ” v5.1).
- Any wasm work (v5.2).
- Vulkano backend or naga (v5.3).
- New capabilities (custom layout, virtualized lists, rich text).
- Splitting `Runner` out of `aetna-wgpu` (defer to v5.3 вЂ” we don't yet know what vulkano needs from a shared runner).
- `String`в†’integer id interning (separate concern; defer unless profiling demands it).

### Open questions for the implementing instance

1. **Should `UiState` itself be decomposed?** Today it holds pointer state + focus state + scroll state + hotkey list + animation map. A `PointerState` / `FocusState` / `ScrollState` / `AnimationState` decomposition would be cleaner. **Recommend:** defer. Pre-committing to the split before seeing how the side-map refactor lands is premature.
2. **Should `computed_id` become an integer?** Real perf win, minor API change. **Recommend:** no, in v5.0. Bundle with a future profile-driven slice.
3. **Where does `Color` live?** Tree-related (used by `El.fill`) but also referenced by anim and draw_ops. **Recommend:** `tree/color.rs` for proximity to its primary user; re-export from `lib.rs` so callers don't care about the path.

## What v5.0 does *not* commit us to

- The v5.1+ slices are still negotiable. If text decoupling reveals a fundamentally different shape we like better, the v5.0 boundaries can flex. The plan above is the cleanest starting point given what we know.
- `Painter` as a *trait* is still deferred. v5.0 ships `WgpuPainter` as a concrete type. If v5.3 finds a real need for a trait, we add one then.
- The `attempts/attempt_4/` directory is untouched. v0.4 remains a working reference until v5.4 completes вЂ” we can A/B against it any time.
